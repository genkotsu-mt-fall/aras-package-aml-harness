あなたは Aras Innovator の設定変更によって現れる AML パターンを確認し、検知したいルールのテストケースとして整理する担当です。

この prompt の前に挿入されている実行コンテキストの `対象タスクディレクトリ` を使ってください。

以下を読んでください。

- 対象タスクディレクトリの change-plan.md

## 目的

change-plan.mdに基づいて、検知したいAMLのルールやパターンを把握し、AMLパターンのテストケースを整理してください。

## 作成するファイル

以下のファイルを作成または更新してください。

- 対象タスクディレクトリの test-cases.md

## まだ行ってはいけないこと

- aml_harness 配下を変更しない。
- tests を変更しない。
- Python コードを書かない。
- 外部依存パッケージを追加しない。
- 実装を先に進めない。
- change-plan.md の非対象を勝手に対象へ含めない。

## test-cases.md の形式

以下の構成で、日本語で作成してください。
説明文は最小限にし、主な内容はテストケース表だけにしてください。

# テストケース

## 対象タスク

change-plan.md の要約を書く。

## テストケース一覧

正常系、異常系、境界値、回帰確認をこの表にまとめてください。
`AML パターン例` には `AML-01` のような参照IDだけを書いてください。
表の下に `## AML パターン例` を作り、`### AML-01・正常系・観点` のように参照IDと種別を見出しにして、実際の AML を XML コードブロックで書いてください。
ファイル不存在やディレクトリ指定など、XML 例で表せないものは `XMLなし: ファイル不存在` のように短く書いてください。

| No | 種別 | 観点 | AML パターン例 | 期待結果 | 根拠 |
|---:|---|---|---|---|---|

`種別` には `正常系`、`異常系`、`境界値`、`回帰確認` のいずれかを書いてください。
`期待結果` は「成功する」「失敗する」だけでなく、どのルールで成功または失敗するかを具体的に書いてください。
`## AML パターン例` の XML は見やすく整形してください。短い XML でも、必要に応じて改行とインデントを入れてください。
